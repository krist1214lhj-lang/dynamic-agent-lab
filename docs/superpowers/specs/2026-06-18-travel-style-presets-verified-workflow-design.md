# 여행 방식 프리셋 + 검증된 워크플로우 설계

작성일: 2026-06-18
상태: 승인됨 (구현 계획 대기)

## 1. 배경 / 문제

현재 앱은 "포함할 정보(체크박스) + 추가 조건(동행인/테마/우선순위) + 특별한 취향(자유입력)" 구조로 여정을 설계한다. 한계:

- 사용자의 **여행 방법(형태·이동수단)**을 고를 축이 없다. 이동수단은 교통/예산 결과를 좌우하는데 입력이 없다.
- 테마가 5개(힐링·액티비티·맛집탐방·사진·문화)로 제한적이다.
- 결과에 실데이터(외부 API)와 mock/fallback이 **구분 없이** 섞여 나와 신뢰하기 어렵다. (예: 서비스키 없을 때 mock이 그대로 노출.)

## 2. 목표 / 비목표

**목표**
- 여행 **형태**와 **이동수단**을 선택 축으로 추가한다.
- 테마를 10개로 확장한다.
- "여행 방식 프리셋"으로 빠르게 시작하고, 확장 패널에서 세부 조정한다. (UI안: C+B 결합)
- 각 선택이 해당 에이전트 동작에 정확히 반영된다.
- 결과를 **검증 상태(✅ 검증 / ⚠ 추정)** 로 구분하고, "검증된 것만 보기" 토글을 제공한다.

**비목표 (v1 제외)**
- 맛집·행사·숙박의 실제 외부 API 연동 (계속 mock, ⚠추정으로 표기).
- 패키지 형태의 묶음 예산 정밀 모델링 (라벨/표기까지만).
- 해외 목적지 확장 (별도 설계 문서).

## 3. UI 구조 (프리셋 C + 세부 패널 B)

1. **여행 방식 프리셋** — 상단 큰 카드 6개 + "직접 설정". 하나 선택 시 아래 패널의 형태/이동수단/테마/페이스를 자동 채움.
2. **세부 조정 패널** — 프리셋 값이 채워진 상태로 자유 수정:
   목적지 · 출발지 · 날짜 · 인원 / 여행 형태 / 이동수단 / 테마(10, 복수) / 동행 / 페이스(빡빡·보통·여유) / 우선순위 / 특별한 취향 / 포함할 정보.
3. **결과** — 카드마다 검증 배지. 상단 "검증된 것만 보기" 토글(기본 OFF = 검증+추정 함께, 추정은 ⚠배지).

## 4. 선택 축 정의

- **여행 형태(travel_format)**: 자유여행 · 당일치기 · 캠핑/차박 · 기차여행 · 패키지
- **이동수단(transport_mode)**: 자가용 · 기차/KTX · 항공 · 대중교통 · 렌터카
- **페이스(pace)**: 빡빡 · 보통 · 여유
- **테마(themes, 10)**: 힐링 · 액티비티 · 맛집탐방 · 사진 · 문화 · 자연/풍경 · 휴양/온천·스파 · 축제/이벤트 · 가족·키즈 · 미식 심화

### 테마 정리(중복 해소)
- **가족·키즈** 테마 선택 시 동행 '아이와 함께'를 자동 포함하고 키즈 친화 보정을 켠다(별도 중복 입력 제거).
- **미식 심화**는 베이커리/파인다이닝/미슐랭 강조(food 키워드 상향 + 음식 예산↑). **맛집탐방**은 로컬 맛집 위주로 구분.

## 5. 워크플로우 매핑 (조건 → 에이전트)

신규 입력은 `WorkflowRequest`에 추가하고 `input_data`로 주입한다: `travel_format`, `transport_mode`, `pace` (테마는 기존 `additional_conditions.themes` 활용).

- **여행 형태**
  - 당일치기 → `days=1`, 숙박 에이전트 미선택/0건(기존 동작 재사용).
  - 캠핑/차박 → 숙박 에이전트가 캠핑장/오토캠핑 항목 반환, 숙박 예산 단가↓.
  - 기차여행 → 교통 에이전트 철도 우선, 자가용 제외.
  - 패키지 → v1: 교통·숙박 카드에 "패키지 포함" 라벨 표기(정밀 묶음예산은 비목표).
  - 자유여행 → 기본.
- **이동수단(transport_mode)** → 교통 에이전트가 해당 수단 우선/제한, `budget_model`이 수단별 교통비 반영(기차/항공/자가용·렌터카/대중교통 단가 분기).
- **테마** → 관광/맛집/행사 키워드 힌트 + 예산 배수 + 숙박 스타일. 신규 5테마 매핑 추가.
- **페이스(pace)** → 일정 에이전트의 하루 `time_blocks` 개수 조절(빡빡=많음, 여유=적음).

## 6. 검증 게이트 (실데이터 + 정합성)

각 에이전트 결과에 `verification` 상태를 부여한다: `verified` | `estimated`.

**판정 규칙**
- 실API 보유(destination/tour=tour_api, weather=kma_api, transport=odsay_api): `data_source`가 실제 API면 후보 `verified`, mock/fallback이면 `estimated`.
- 계산형(budget/schedule/planning): 외부 API가 없는 수식 계산 → 정합성 검사 통과 시 `verified`.
- mock 전용(food/event/lodging): 현재 항상 `estimated`.

**정합성 검사(공통, 통과해야 verified 유지)**
- 목적지가 입력 목적지와 일치
- 일정 일수 == 요청 일수
- 예산 total > 0 이고 항목 합과 일치
- 날씨 location == 목적지
- 교통 출발/도착이 origin/destination과 일치
- 일정이 실제 생성된 항목을 참조(빈 참조 없음)

하나라도 실패하면 해당 카드는 `estimated`로 강등하고 실패 사유를 `validation_report`에 기록한다.

**출력 동작**
- 응답의 각 `agent_results[i]`에 `verification`과 사유를 포함.
- UI: `verified`는 정상 표시, `estimated`는 "⚠ 추정치(실데이터 아님)" 배지.
- 상단 토글 "검증된 것만 보기"(기본 OFF). ON이면 `estimated` 카드 숨김.

## 7. 데이터 모델 / 영향 파일

- `main.py`
  - `WorkflowRequest`: `travel_format`, `transport_mode`, `pace` 추가.
  - `run_workflow`: 신규 필드를 `input_data`에 주입. 형태→days/에이전트 선택 보정.
  - 검증 게이트: `validators/travel_validator.py`의 `validate_and_correct`를 확장하거나 신규 `verify_results()`를 추가해 각 결과에 `verification` 부여.
- `budget_model.py`: `transport_mode`별 교통 단가 분기(기존 거리기반에 수단 계수 결합).
- 에이전트: `travel_transport_agent`(수단 우선/제한), `travel_lodging_agent`(캠핑 형태), `travel_schedule_agent`(pace→time_blocks), `travel_food_agent`(미식심화 키워드), `travel_tour_agent`(신규 테마 키워드), `travel_event_agent`(축제 테마).
- `static/index.html`: 프리셋 카드, 세부 패널(형태·이동수단·테마10·페이스), 결과 검증 배지 + 토글, `runWorkflow` payload 확장.
- `validators/travel_validator.py`: 정합성 검사 규칙.

## 8. 테스트 전략

- **단위**: 신규 파라미터 전달(`travel_format`/`transport_mode`/`pace`)이 `input_data`에 반영. budget_model 수단별 교통비 차등. 페이스→time_blocks 수. 캠핑→숙박 캠핑 항목·당일치기→숙박 0. 검증 판정(실API=verified, mock=estimated, 정합성 실패 시 강등).
- **정합성**: 목적지/일수/예산 불일치를 일부러 만들어 `estimated`로 강등되는지.
- **smoke**: 기존 15개 유지 + 신규 필드 기본값 하위호환.
- **브라우저 E2E**: 프리셋 선택→패널 자동채움→여정 그리기→검증 배지/토글 동작, 추천↔상세 일치 유지.

## 9. 단계화(plan에서 구체화)

- Phase 1: 데이터 모델 + 워크플로우 매핑(형태/이동수단/pace) + budget_model 수단 분기 + 에이전트 반영.
- Phase 2: 검증 게이트(verification 상태 + 정합성 검사) + 응답 필드.
- Phase 3: UI(프리셋·패널·배지·토글) + E2E 검증.

## 10. 하위호환

신규 필드 기본값: `travel_format="자유여행"`, `pace="보통"`, `transport_mode=None`. `transport_mode`가 `None`이면 `budget_model`은 수단 계수 없이 **현행 거리기반 교통비**를 그대로 사용하고 교통 에이전트도 수단 제한을 걸지 않는다. 따라서 신규 필드를 보내지 않는 기존 `/run-workflow`·smoke 호출과 완전히 호환된다. 검증 게이트는 응답에 `verification` 필드를 추가할 뿐 기존 카드 렌더를 깨지 않는다(필드 없으면 UI는 verified로 간주).
